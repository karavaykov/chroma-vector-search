# Микросервисная архитектура для Chroma Vector Search

## Текущее состояние (Монолит)
- **Файл:** `chroma_simple_server.py` (901 строк)
- **Протокол:** TCP с кастомными командами
- **Функции:** Индексация, поиск, статистика в одном процессе
- **Проблемы:** Сложность масштабирования, высокое потребление памяти, нет разделения ответственности

## Целевая архитектура (Микросервисы)

### 1. Indexing Service
**Назначение:** Обработка и индексация файлов кодовой базы
**Эндпоинты:**
- `POST /index` - запуск индексации
- `GET /index/status/{job_id}` - статус индексации
- `DELETE /index/{collection}` - удаление индекса

**Функции:**
- Потоковая обработка файлов
- Извлечение метаданных (1С/BSL)
- Генерация эмбеддингов
- Сохранение в ChromaDB

### 2. Search Service
**Назначение:** Семантический поиск по индексированным данным
**Эндпоинты:**
- `POST /search` - семантический поиск
- `GET /search/similar/{chunk_id}` - поиск похожих фрагментов
- `GET /search/metadata` - поиск по метаданным

**Функции:**
- Поиск по векторным эмбеддингам
- Фильтрация по метаданным
- Ранжирование результатов
- Кэширование запросов

### 3. Metadata Service
**Назначение:** Управление метаданными и статистикой
**Эндпоинты:**
- `GET /metadata/stats` - статистика коллекции
- `GET /metadata/files` - список проиндексированных файлов
- `POST /metadata/update` - обновление метаданных

**Функции:**
- Хранение метаданных в PostgreSQL
- Агрегация статистики
- Управление версиями индексов

### 4. API Gateway
**Назначение:** Единая точка входа для клиентов
**Эндпоинты:**
- `POST /api/v1/search` - прокси к Search Service
- `POST /api/v1/index` - прокси к Indexing Service
- `GET /api/v1/health` - проверка здоровья всех сервисов

**Функции:**
- Аутентификация/авторизация
- Логирование запросов
- Балансировка нагрузки
- Rate limiting

## Стек технологий

### Backend
- **FastAPI** - REST API фреймворк
- **ChromaDB** - векторная база данных
- **PostgreSQL** - реляционная БД для метаданных
- **Redis** - кэширование и очереди
- **SentenceTransformers** - модели эмбеддингов

### Инфраструктура
- **Docker** - контейнеризация
- **Docker Compose** - оркестрация
- **Nginx** - reverse proxy (опционально)
- **Prometheus/Grafana** - мониторинг

## Миграция с TCP на REST

### Текущий TCP протокол:
```python
# Команды:
SEARCH|query|n_results
INDEX|file_patterns
STATS
PING
```

### Новый REST API:
```python
# Поиск
POST /api/v1/search
{
  "query": "database connection",
  "n_results": 5,
  "filters": {"language": "python"}
}

# Индексация
POST /api/v1/index
{
  "file_patterns": ["**/*.java", "**/*.py"],
  "max_file_size_mb": 10
}

# Статистика
GET /api/v1/stats

# Здоровье
GET /api/v1/health
```

3. Обновить документацию

### Этап 4: Мониторинг и оптимизация

2. Настроить Grafana дашборды
3. Оптимизация производительности

## Преимущества новой архитектуры

1. **Масштабируемость:** Каждый сервис можно масштабировать независимо
2. **Отказоустойчивость:** Падение одного сервиса не влияет на другие
3. **Поддержка:** Легче добавлять новые функции
4. **Мониторинг:** Индивидуальные метрики для каждого сервиса
5. **Развертывание:** Независимые циклы релиза

## Оценка сложности

- **Indexing Service:** Средняя (миграция существующей логики)
- **Search Service:** Низкая (простая обертка над ChromaDB)
- **Metadata Service:** Средняя (новая функциональность)
- **API Gateway:** Низкая (проксирование запросов)
